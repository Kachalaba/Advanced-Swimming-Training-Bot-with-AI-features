# План разработки Android-приложения SPRINT AI

## 1. Текущее состояние кодовой базы

Проект уже частично готов к мобильному клиенту — это главный вывод аудита:

| Слой | Что есть | Готовность для Android |
|------|----------|------------------------|
| Ядро анализа (`video_analysis/`) | YOLOv8 + MediaPipe, 40+ метрик плавания, бег, вело, реабилитация | Готово, не зависит от UI |
| REST API (`backend/`) | FastAPI: плавание, бег, реабилитация, атлеты, инструменты; SSE-прогресс; job registry | Основа готова, нужны доработки (см. фазу 0) |
| Веб-клиент (`frontend/`) | Next.js / React + TypeScript | Референс UX и API-контрактов |
| Streamlit (`app.py`, `pages/`) | Полнофункциональный UI | Остаётся как внутренний/десктопный инструмент |

**Вывод:** не нужно «портировать Python на Android». Правильная архитектура —
**тонкий Android-клиент + существующий FastAPI-бэкенд**. Тяжёлый CV/ML-анализ
(YOLO, MediaPipe) остаётся на сервере; телефон снимает/выбирает видео,
загружает его, показывает прогресс и результаты.

## 2. Выполненный рефакторинг (этот PR)

`video_analysis/analyzer_factory.py` был жёстко завязан на `streamlit`
(`@st.cache_resource`), из-за чего бэкенд (где Streamlit не установлен —
см. `backend/requirements.txt`) не мог использовать фабрику и создавал
анализаторы напрямую, нарушая паттерн из CLAUDE.md.

Сделано:

- Фабрика стала **Streamlit-опциональной**: внутри запущенного Streamlit
  используется `st.cache_resource`, вне его (FastAPI, CLI, тесты) —
  потокобезопасный in-process кэш с теми же семантиками.
- Добавлены юнит-тесты `tests/unit/test_analyzer_factory.py`
  (кэширование, потокобезопасность, работа без установленного Streamlit).

Замечание: в `backend/app/services/` анализаторы с мутабельным состоянием
(`RunningAnalyzer`, `RehabAnalyzer`) намеренно создаются по одному на job —
общий кэшированный экземпляр нельзя использовать из параллельных задач.
Это ограничение задокументировано в docstring фабрики.

## 3. Выбор технологии клиента

| Вариант | Плюсы | Минусы | Вердикт |
|---------|-------|--------|---------|
| **Kotlin + Jetpack Compose** (рекомендуется) | Лучшая работа с камерой (CameraX), видео (Media3/ExoPlayer), фоновой загрузкой (WorkManager); нативный UX; прямой путь к on-device MediaPipe | Отдельная кодовая база | ✅ Основной вариант |
| React Native | Реюз TypeScript-логики и API-клиента из `frontend/` | Слабее с камерой/видео/фоновыми задачами | Запасной, если важна скорость старта |
| Flutter | Быстрая разработка UI | Нет реюза существующего кода | Нет явных преимуществ здесь |
| Python на Android (Kivy/Chaquopy) | «Один язык» | YOLO/MediaPipe/torch на телефоне: размер, перегрев, поддержка | ❌ Не рассматривать |

## 4. Фазы разработки

### Фаза 0 — Подготовка бэкенда (1–2 недели)

Цель: API, к которому можно безопасно подключить мобильный клиент из интернета.

- [x] Развязать `analyzer_factory` со Streamlit (этот PR)
- [ ] Аутентификация: JWT (вход по email/код), middleware в FastAPI
- [ ] Версионирование API: префикс `/api/v1` (текущий `/api` оставить для веб-фронта на переходный период)
- [ ] Персистентный job registry: текущий `backend/app/services/jobs.py` —
  in-memory, single-process; перейти на SQLite/Redis-backed очередь, чтобы
  задачи переживали рестарт и работали при нескольких воркерах
- [ ] Недостающие endpoint'ы: велосипед (`cycling_analyzer`), сухое плавание
  (`exercise_analyzer`), AI-чат (`ai_chat.py`) — по образцу `analysis.py`
- [ ] Хранение видео: вынести из локальной ФС в объектное хранилище (S3/R2) или
  как минимум настроить `SESSION_VIDEO_DIR` на примонтированный том
- [ ] Деплой: Docker (уже есть `backend/Dockerfile`) + VPS/облако с доменом и TLS;
  GPU опционально (MediaPipe работает на CPU, YOLO желателен GPU)
- [ ] Лимиты: rate limiting, уже есть валидация и лимит размера загрузки —
  проверить значение для мобильных сетей

### Фаза 1 — MVP Android (4–6 недель)

Стек: Kotlin, Jetpack Compose, Retrofit + OkHttp (SSE через `okhttp-sse`),
Media3/ExoPlayer, CameraX, WorkManager, Hilt.

Экраны и маппинг на существующее API:

| Экран | Endpoint |
|-------|----------|
| Выбор/запись видео → загрузка | `POST /api/analysis/swimming`, `POST /api/analysis/running` |
| Прогресс анализа | `GET /api/analysis/{sport}/{job_id}/events` (SSE) + polling-fallback `GET /api/analysis/{sport}/{job_id}` |
| Результаты: метрики, оценки | `result` из job status |
| Аннотированное видео | `GET /api/analysis/{sport}/{job_id}/video` (стрим в ExoPlayer) |
| Сохранение сессии | `POST /api/analysis/swimming/{job_id}/save` |
| Атлеты и история | `GET /api/athletes/...` |

Ключевые мобильные нюансы:

- **Сжатие видео на устройстве** перед загрузкой (Media3 Transformer:
  720p/30fps достаточно для пайплайна) — иначе загрузка по LTE будет болью
- **WorkManager** для загрузки: переживает сворачивание приложения и обрыв сети
- SSE с авто-reconnect; при обрыве — переход на polling статуса (endpoint уже есть)

### Фаза 2 — Полнота функций (3–4 недели)

- AI-чат с тренером (endpoint из фазы 0; стриминг ответов Claude)
- История и графики прогресса (Vico/Compose Charts) по данным `athletes` API
- i18n: украинский + английский (ключи переиспользовать из `i18n/translations.py` —
  выгрузить в общий JSON, чтобы веб и Android делили словарь)
- Офлайн-кэш результатов: Room + локальное хранение последних отчётов
- Push-уведомление «анализ готов» (FCM) — полезно для длинных видео

### Фаза 3 — On-device live-режим (4+ недели, опционально)

- Live-реабилитация без сервера: **MediaPipe Tasks Vision (Pose Landmarker)
  для Android** + порт правил из `rehab_analyzer.py` на Kotlin
  (углы суставов, ROM, подсчёт повторений — чистая геометрия, портируется легко)
- Это единственное место, где логика дублируется на Kotlin; зафиксировать
  правила в JSON-спецификации, чтобы Python и Kotlin читали одни пороги
  (источник — `video_analysis/constants.py`)

### Фаза 4 — Релиз (2 недели)

- CI/CD: GitHub Actions → сборка AAB, подпись, выкладка в внутренний трек Play Console
- Sentry Android SDK (DSN уже поддерживается в проекте)
- Play Store: приватность (видео людей! нужна политика конфиденциальности,
  согласие на обработку), возрастной рейтинг, скриншоты
- Бета через внутреннее/закрытое тестирование с тренерами

## 5. Риски и решения

| Риск | Решение |
|------|---------|
| In-memory jobs теряются при рестарте сервера | Фаза 0: персистентная очередь |
| Большие видео по мобильной сети | Сжатие на устройстве + лимит длительности ролика |
| Долгий анализ (минуты) при свёрнутом приложении | WorkManager + FCM-уведомление о готовности |
| Стоимость GPU-сервера | Старт на CPU (MediaPipe ок), очередь сглаживает пики; GPU позже |
| Расхождение порогов метрик между Python и Kotlin (фаза 3) | Единая JSON-спека констант из `constants.py` |

## 6. Итоговая последовательность

```
Фаза 0 (бэкенд)  ──►  Фаза 1 (MVP)  ──►  Фаза 2 (функции)  ──►  Фаза 4 (релиз)
                                              │
                                              └──►  Фаза 3 (on-device live) — параллельно/опционально
```

Ориентировочно 3–4 месяца до публичной беты силами 1–2 разработчиков
(1 backend/Python + 1 Android/Kotlin).
